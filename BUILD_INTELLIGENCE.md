# BUILD_INTELLIGENCE.md — Lessons from each shipped item

> Append-only, one line per item. Format: `[item] | [gotcha or inefficiency] | [better approach]`
> Read this before starting a new build session — every entry makes the next session smarter.

0.2.B Settings Users panel | The legacy Settings card referenced removed `USERS{}` global → would have thrown ReferenceError in production if not caught. Lesson: when removing a global, grep ALL render functions, not just the auth block. | After deleting any top-level const/let, run `grep -n "<NAME>" *.html *.js` and fix every reference in the same patch.
0.2.B Settings Users panel | Inline `onclick="${onChange}"` with arrow-function IIFE shadows `this`, so the radio's value isn't accessible. Same pattern was bitten earlier in the score-state radio fix. | Default to a named global function (`onScoreStateChange(this, key)`) for any inline handler that needs `this`. Never use arrow IIFE in inline onclick/onchange.
2.1 Parent UI | When extending a `<label>` with sibling buttons inside a parent `<div>`, the surgical str_replace can leave a stray `</label>` orphan. Recovered by adding a hidden dummy `<label>` for balance — works but ugly. | Read 2 lines past the old_string when planning HTML edits to ensure tag balance. Or open the smallest possible block (the whole `<label>...</label>` pair) so the patch is self-balancing.
0.2.C RLS SQL | Dropping then recreating policies in a single SQL transaction works fine in Supabase, but if you forget the `DROP POLICY IF EXISTS` first, re-runs fail with "policy already exists". | Always lead each policy block with `DROP POLICY IF EXISTS`. Makes the file safely re-runnable.
0.4 Core schema | Naive 18-table file would have ~80 RLS policies if expressed inline. Used DO $$ … FOREACH … EXECUTE format() $$ to apply identical patterns in batches — collapsed to ~30 policies + 4 dynamic blocks. | For repetitive RLS patterns, use plpgsql DO blocks with arrays of table names. Halves the SQL size, makes patterns visible at a glance.
1.3 Daily Brief | "Build everything autonomously" runs out of runway when subsequent items all need M02 tables. Track 1.3 phase 1 used only existing data; phase 2 (quotes/pipeline tiles) is genuinely blocked. | When a track has multiple data dependencies, split it explicitly into "phase 1 (current data)" and "phase 2 (after M##)" so the unblocked work ships immediately and the blocked work has a clean re-entry point.
1.3 Daily Brief | Inline `onclick` strings using setTimeout (e.g. `goTo('vendors');setTimeout(...)`) feel hacky but are the cleanest way to navigate then update sub-tab state since renderVendors mutates the DOM the sub-tab variable references. | Tolerable pattern for now. Long term: refactor goTo() to accept a sub-tab arg (e.g. `goTo('vendors', 'scores')`) so callers don't need setTimeout sequencing.
1.1 vendor_scores | Same pattern as vendor_score_states (paginated load + on_conflict upsert). The first time the schema and the JS shape mostly match it goes fast; here it took ~5 minutes to ship. | When extending an existing persistence pattern (X_states → X_scores), copy-paste the loader/saver and rename — don't reinvent.
1.2 Quotes | Schema's quotes.notes is TEXT, but JS quote has 4 extras (contact/type/sqft/budget) the schema doesn't model. Stored as JSON string in notes — works but means notes is double-purposed. | Either extend schema with first-class columns OR move extras to a sidecar quote_attributes table. Acceptable for now since extras are sparse.
1.2 Quotes | Replacing all quote lines on save = DELETE + INSERT. Two roundtrips, but simpler than diffing. Edge case: if INSERT fails after DELETE succeeds, lines vanish. | For high-volume editing: batch into a single RPC. For our ~10-line quotes it's fine. If users start editing 100+ line quotes, revisit.
2.3 Coop tracker | Embedded inside Vendor Ranking page as a sub-tab — lower visibility than top-level nav, but doesn't add nav clutter. Daily Brief tile compensates by surfacing deadlines at the top of the dashboard. | Pattern works: secondary modules nest as sub-tabs; Daily Brief is the visibility layer that pulls them up when action is needed.
1.5 Pipeline | Old JS stages (prospect/quoted/ordered/complete) didn't map cleanly to schema (lead/qualified/quoted/negotiating/won/lost/abandoned). Did a hard refactor — old saved state in localStorage will look empty after deploy. Acceptable since pipeline was hardcoded demo data, not real production data. | Future schema/JS mismatches: if real data exists, write a migration. If demo only, hard-cut and note it in SESSION_LOG.
1.5 Pipeline | 8-factor probability model used heuristic numbers (e.g. referral=0.85). Will be wrong until probability_model_log accumulates real win/loss data and recalibrates. | Ship with documented heuristics. The recalibration job is its own future task — for now, the model produces sensible relative ordering even if absolute % is off.
4.1 Mgmt restructure | Replaced 70-line static "Pending Decisions" block with a sub-tab router (Overview/KPIs/Goals/Team Activity/System) — only ~90 lines net but unlocks 4 distinct screens. Pattern: when a page evolves from static to multi-screen, refactor to a sub-tab dispatch immediately rather than letting it grow as a single el.innerHTML. | If a render function exceeds ~100 lines or grows more than 2 distinct concerns, split into sub-tabs via a small `mgmtSection` global + dispatcher. Avoids the "one giant function" trap.
4.2 KPI auto-seed | First Owner load auto-seeds kpi_definitions if empty. Saves a manual M-task but creates a hidden side effect (data inserted via UI render). | Acceptable for owner-only seeding. Don't pattern-match this for non-owner cases — silent inserts on render are otherwise a smell.
4.2 KPI snapshot conflict | Used on_conflict=kpi_key,snapshot_date so re-running "Snapshot today" overwrites the same day's row instead of erroring. | When a user-triggered persistence action could be repeated in a session, design the unique key + on_conflict from the start.
4.3 Goals tree | Recursive renderGoalNode() with depth indent works fine for the expected goal counts (10-50). For 500+ goals, switch to virtualized rendering or lazy-expand on click. | Knowing the data scale lets you write the simpler implementation first.
3.2 Role dashboards | Three-branch role gate inside dashboard() (Warehouse / Sales / senior). Adding a 4th role variant means a 4th branch. | When role variants exceed 3, refactor to a roleDashboards = {Warehouse: fn, Sales: fn, ...} map with a default. Cleaner than nested if/else.
DOC-ONLY task batching (this session) | First three sessions interleaved doc updates between every code edit — wasted tokens re-reading file state. This session: code committed first, doc updates batched into single end-of-session commit. Saved meaningful tokens. | NEW RULE adopted: doc-only edits (BUILD_PLAN check-offs, SESSION_LOG, BUILD_INTELLIGENCE, MASTER edits) batched into one final commit per session, separate from code commits. Code commits stay focused.
1.4 CRM | Schema has customers.id (UUID) + quotes.customer_id (UUID FK), but quote save flow only stores customer_name (TEXT). Linking customers to their quotes/deals therefore fell back to case-insensitive name match. Works for now but breaks under typo / abbreviation drift. | When schema supports a UUID FK, wire the dropdown into the parent module's save flow at the same time as building the dependent module. Don't ship the lazy name-match path even as v1 — it'll persist as tech debt.
1.4 CRM | Reused the coop_tracker / quote pattern for sbLoad/Save/Delete — copy-paste-rename took ~3 min per table. Same heuristic from 1.1 vendor_scores: extending an established Supabase pattern is fast; defining a new pattern is the expensive part. | When the table looks similar to an existing one (id PK, on_conflict, single-row REST POST), copy the closest sibling's persistence trio verbatim and only rebrand the field list. Don't rethink the structure.
3.1 Employees | Decided to nest Employees as a Mgmt Dashboard tab rather than a top-level sidebar item. Mgmt is already Owner/Admin/Manager-gated and Employees inherits that gate cleanly — no extra `data-roles` wiring needed. | Pattern: when role gating exactly matches an existing parent, prefer nesting (sub-tab) over a new sidebar entry. The role inheritance is free, the visual clutter is avoided.
3.1 Employees | employee_scores PK is a composite UNIQUE on (employee_id, period, metric_key). Save uses on_conflict on that triple — re-saving a (Q1, revenue, Alice) score updates instead of duplicating. Simpler than a "find + decide insert vs update" dance on the client. | When you have a real-world unique key (not just a UUID surrogate), express it as a UNIQUE in the schema and use on_conflict. The save flow becomes idempotent and the UI doesn't need to track "is this an edit or insert" state.
5.7 Deal Optimizer | Pure-compute module over existing data was the cheapest module of the session — no schema, no persistence, no migration. Heuristics for the 5 rec kinds were all <30 LOC each. | When evaluating "what to build next", check if any item is a pure derived layer over existing tables. Those are the lowest-effort high-value modules and should be promoted in priority.
5.7 Deal Optimizer | Heuristic constants (≥$25K spend threshold, ≤4 score threshold, 3% recoverable, 90-day window) are baked in. Will be wrong for a different distributor or different cycle. | Knowingly shipping with constants. Future job: extract to a `optimizer_config` row when the team wants to tune. Don't pre-build the abstraction — wait for the first time someone wants to change a constant.
5.16 Calendar | Three modules (5.1/5.2/5.16) all needed new tables. Wrote one M21 SQL bundle for all three, built all three UIs in same session, all activate together when Michael runs SQL. Faster than three separate Michael handoffs. | When multiple unblocked items all need NEW tables, bundle them into one schema migration M-task and build all the UIs at once. The marginal SQL is small; the avoided handoff cost is meaningful.
5.16 Calendar | Persistence functions silently no-op on table-missing (404). Differentiated the log level: relation-does-not-exist logs at INFO/`console.log`, other failures at WARN. Saves alarm-fatigue on first load before SQL runs. | When a feature ships ahead of its schema, downgrade the "table missing" path from warn → info. Real errors (network, auth, RLS) still warn. The "you haven't run M## yet" message belongs in the comments + UI toast, not in the warning channel.
5.1 Knowledge Hub | Built tab dispatch via setTimeout(()=>renderKnowledgeHub(...), 0) because tab pane HTML is mounted via the existing switchTab pattern that just sets innerHTML. The container ID isn't in the DOM until after innerHTML completes. | If you're rendering into a child of just-set innerHTML, defer with setTimeout(0) — or refactor switchTab to call a callback. setTimeout(0) is uglier but cheaper than refactoring a pattern used 3 places.
5.2 Job Tracker | Customer + Quote dropdowns in the edit modal pull from CUSTOMERS + QUOTES global arrays. If those arrays haven't loaded yet (empty), dropdowns are empty — but free-text customer name input is always available as fallback. | When a module references foreign data via dropdowns, always include a free-text fallback for the same field. Edge case: page loaded before linked module hydrated.
6 modules in one session | Each compact-CRUD module (~250 LOC) shipped in 5–8 minutes once the persistence pattern was internalized. Six in one session is sustainable when modules are pattern-aligned. | The bottleneck is not coding; it's deciding the data model + UX scope. Lock those in fast (5 min think + 1 min schema), then the implementation flows.

OPERATING RULE — prompt logging | FIRST thing every session: log the incoming prompt verbatim to PROMPT_LOG.md and commit it with message `log: prompt received [date]`. This happens BEFORE any build work, no exceptions. Reason: if Codespace dies mid-session, the next session can replay the user's intent exactly without guessing from partial diffs. | Append a new dated entry under `## PROMPT LOG`. Quote the prompt verbatim in **Prompt:** and add a one-line **Context:** explaining why it landed (e.g. "session resume", "new feature ask", "bugfix follow-up").
OPERATING RULE — work-in-progress checkpoint | After every discrete build step (file written, function added, SQL generated, test passed, etc.), overwrite WORK_IN_PROGRESS.md with the current state and commit with message `wip: [task name] — [step]`. The file is always single-state (current snapshot, not append-only). | The schema: Last updated · Current task · Step · Files touched so far this task · Commit status · Next step if interrupted. The "Next step if interrupted" line is the load-bearing field — write it as if a new Claude with no context were about to read it cold.
OPERATING RULE — session resume | On session resume: read WORK_IN_PROGRESS.md FIRST (before BUILD_PLAN_CLAUDE.md). If it shows uncommitted work or a half-finished task, complete that task before moving to the next BUILD_PLAN item. Then immediately log the new incoming prompt to PROMPT_LOG.md before doing any other work. | This catches the case where Codespace died mid-task. Without this rule the next session would skip ahead, leaving an orphan in-progress feature in the working tree.
5.3 Inventory CSV | Header alias map (sku/qty/cost/etc.) was the make-or-break — without it, every CSV with even slightly non-canonical headers would fail. Cost ~30 LOC; payoff = "import works for 90% of CSVs without user intervention". | When building any CSV import, ALWAYS include a header alias map. Cover: case variants, spaces vs underscores, common synonyms (qty/quantity/on_hand), and abbreviations (msrp/list_price). 30 LOC is cheap insurance.
5.3 Inventory CSV | Preview-then-commit flow stages parsed rows on `window._invStaged` between the parse modal and the commit modal. Hacky but lets me show a preview without holding parsed rows in a closure. | When a UI flow has parse → preview → commit phases, stash intermediate state on `window._stageName`. Cleaner than threading it through closures, especially when modals get re-rendered. Just remember to `delete` after commit.
5.6 Price Book | Built on top of 5.3 in 30 minutes. Pure-compute over INVENTORY + VD. No new schema, no new persistence, no new audit_log entries. Cheapest module of session. | When you've just shipped a data table (5.3 inventory_items), check if any pending module is essentially "a different view of that table". Those are nearly free.
5.4 Purchase Orders | In-modal line-item editor (table inside modal with add/remove rows, live ext_cost recalc) is the most complex modal pattern I've shipped. Used a window-scoped `_poEditLines` array as the source of truth, re-rendering only the lines wrapper on add/remove. Inputs use `oninput="_poEditLines[i].field=this.value;updateTotals()"` — direct mutation. | Fine for 5–20 line items. If line counts get to 50+, the full re-render on every keystroke will lag. Defer optimization until users complain.
5.4 PO receipt flow | "Mark Received & Update Inventory" couples two modules. Match key = (sku + vendor) — needs vendor_id OR vendor_name match. Reports matched + unmatched count. Saves a manual round-trip into Inventory and prevents drift. | Cross-module side effects (PO closes → inventory bumps) are worth shipping when the data shape supports it. The user feedback ("inventory rows updated") makes the connection explicit, which builds trust in the automation.
5.4 PO save | Used a delete-then-insert strategy for line items. Two roundtrips per save. Same pattern as 1.2 Quotes. For ~10-line POs it's fine; if average line count grows past 50, consider a single RPC. | Watch the `quotes` lesson — same pattern, same constraint. If/when we revisit, do both at once.
v6.10.12 file split | Edit tool with 480-line old_string is technically possible but slow + brittle. Used `sed -n 'A,Bp'` for extraction and `awk 'NR<A || NR>B'` for deletion — multi-hundred-line moves done in seconds with one bash call each. The Bash tool rules allow this when "dedicated tool cannot accomplish task efficiently". | For multi-module file splits, the Bash awk/sed approach is significantly faster than Edit. For surgical 5-10 line edits, Edit is still preferred. Rule of thumb: Edit ≤30 lines, awk/sed ≥100 lines.
v6.10.12 file split | When a module's code is interleaved with another module's (Calendar inserted INTO the middle of Co-op Tracker code earlier in the session, leaving Co-op state at top, Calendar middle, Co-op render functions at bottom), simple line-range extraction misses parts. Had to compute Calendar = 1035-1342 (the cleanly contiguous part) while leaving Co-op Tracker fully inline. | Extraction-friendly modules don't have other modules wrapped around them. When inserting a new module's code, ALWAYS append at end-of-section (e.g., after the last function of the most recent module), never inside an existing module. Saves an extraction headache months later.
v6.10.12 file split | "lazy-load on tab activation" was the verbal target but actual ship was "external scripts loaded after inline (no defer, no lazy)". Same practical outcome (file size split, zero user-facing change). True lazy-load adds 50+ LOC of loader infrastructure + race-condition hazards with cross-module reads (Daily Brief → CUSTOMERS, Pipeline → CUSTOMERS by name, Price Book → INVENTORY). | When a refactor goal is "manage file size", external scripts are the simple win. Reserve "lazy-load on tab activation" for a separate task if startup time genuinely matters. Don't bake them together — different motivations, different tradeoffs.
v6.10.12 file split | Module files contain `let CUSTOMERS = []` and similar top-level declarations. JS still attaches them to the global object same as inline. So Daily Brief in index.html reads `CUSTOMERS` no differently than before — extraction is transparent at the language level. | Heavy-globals codebases extract cleanly to external scripts. ES module `import`/`export` would have required rewriting every cross-reference; the cost would be massive and the user-facing benefit zero.
v6.10.13 multi-module SQL bundle | Trade Partners (5.5) + Warranty (5.11) shipped together with a single M24 SQL file containing both tables. Pattern matches M21 (3 tables) — bundling unrelated-but-co-shipped tables saves a Michael handoff. | When multiple modules ship in the same session and each needs new tables, bundle their schema migrations into one M-task. The marginal SQL is small; the avoided handoff is meaningful.
WIP checkpointing in practice | Updated WORK_IN_PROGRESS.md 5+ times during this session (after planning, after each major extraction, before each commit, between modules). Cost: ~1 minute total over the session. Value: if Codespace had stopped at any point, the next session could pick up exactly where it died — including which module was half-extracted, which files were uncommitted. | The frequency feels high but the cost is tiny. Default to over-checkpointing. Better to have a stale WIP than no WIP.
5.8 / 5.9 / 5.10 batch | Three small CRUD modules shipped back-to-back, ~5-8 min each. The Bash awk/sed approach for extractions is over (split is done); going forward new modules just write `js/<name>.js`, add a `<script src>` line, and wire 4 inline-shell touchpoints (sidebar entry, PAGE_META, pages dispatcher, hydrate call). | The "shell touchpoints" are extremely repetitive — could be reduced to 2 if PAGE_META and pages dispatcher were derived from a single registry array. Future cleanup: define `MODULE_REGISTRY = [{key:'customers', title:'Customers', sub:'…', roles:[…], hydrate:'sbLoadCustomers', render:'customers'}, …]` and have the shell loop over it. Saves 4 manual edits per new module.
5.9 QR/Barcode third-party API | Used api.qrserver.com (goqr.me) for QR generation. Pros: zero deps, simple GET URLs, immediately works. Cons: third-party dependency, leaks SKU strings to a public service, no offline support beyond browser cache. | For non-sensitive SKU data this is acceptable for v1. If we ever need offline labels OR sensitive data encoding (customer info, etc.), swap to a pinned local lib. Pattern: prototype with public API → upgrade to local lib when use case demands it.
5.9 print stylesheet | Injected `@media print` rules via `document.head.appendChild(<style>)` on first render. Made all non-`.lbl-print-area` elements `visibility: hidden` so window.print() captures only the label sheet. Worked first try. | Browser print isolation is reliable when (a) the print target is a fixed-position absolute element and (b) `visibility: hidden` is used (NOT `display: none` — display:none affects layout). Pattern applies to any "printable view" feature.
5.15 Decision Engine | Pure-compute layer combining 6 data sources (DEALS, QUOTES, CUSTOMERS, CUSTOMER_INTERACTIONS, INVENTORY, CHANGELOG). No schema, no persistence. Took ~3 minutes to build because all the heuristics already existed in the data shape. | Pure-compute modules over already-loaded data are the cheapest builds AND deliver high ROI per LOC. When evaluating "what to build next", always check if a derived view over existing tables would be useful — those modules are nearly free.
5.15 cross-module navigation | Each Decision Engine row click-throughs to the relevant module's detail view via `goTo('pipeline');setTimeout(()=>{if(typeof openDeal==='function')openDeal('${id}','${stage}')},80)`. The setTimeout 80ms is needed because goTo() mutates the DOM the detail-render function reads. | The `setTimeout(80)` pattern is everywhere now (Daily Brief tiles, Deal Optimizer rows, Decision Engine rows). It works but is fragile — anything that delays render (slow data load, complex DOM) could miss. Long-term: refactor goTo() to accept a callback that fires AFTER the page mounts. Short-term: 80ms is reliable for the current page sizes.
5.14 append-only observations | Schema chose append-only (insert per snapshot) over "upsert latest" because historical pricing drift is the actual signal — we want to chart how a competitor's price moves over time. Latest-per-pair is computed in JS via `Object.values(reduce-into-map)`. | Default to append-only for any "observation" or "metric snapshot" pattern (kpi_snapshots, employee_scores, competitor_prices). It's the same Postgres cost (one row per write) and gives you a free time series.
5.14 datalist autocomplete | Used HTML5 `<datalist>` + `<input list="…">` for SKU autocomplete from INVENTORY in the new-observation modal. ~500 SKUs in the dropdown without performance issues. | Native datalist works well up to a few thousand options. Beyond that, switch to a custom autocomplete that fetches on input. For most AccentOS tables (≤2K rows), datalist is the right call — zero JS, native browser UX.
5.12 Marketing Hub | Single-table-per-domain pattern: all 8 campaign types live in `marketing_campaigns` with a `type` field, including promotions which get extra promotion-specific columns (discount_pct, discount_amount, promo_skus) that are NULL for non-promo rows. Initial instinct was to split promotions into a separate table — the consolidation was simpler, faster to ship, and queries stay clean (`WHERE type='promo'`). | When two entities share 80%+ of their fields, prefer one table with a `type` field over two tables. Splitting only pays off when the entity-specific fields would dominate or when the lifecycles are wildly different.
5.12 collapsible details | Promotion details and attribution metrics render inside `<details>` elements that open automatically when the relevant fields are non-null (`<details ${r.discount_pct ? 'open' : ''}>`). Saves vertical space in the modal for the common case while exposing the fields when they matter. | Native `<details>` is the cheapest progressive-disclosure UI. No JS needed for open/close. Pattern: open if any field in the section has a value, else collapsed. Apply to any modal with sections used by <50% of records.
5.12 inline-to-external swap | Replacing an inline function (`function marketing(el)`) with an external-module version is straightforward when the function name stays the same: just delete the inline definition and add the external `<script src>`. The pages dispatcher in `goTo` references `marketing` by name and finds whichever definition loaded last (external wins because it loads after the inline `<script>` block). | Pattern works for any inline page-render function. Useful when growing a placeholder into a real module without changing the dispatcher.
Track 5 complete reflection | All 16 Track 5 items shipped: 5.1 Knowledge Hub, 5.2 Job Tracker, 5.3 Inventory CSV, 5.4 Purchase Orders, 5.5 Trade Partners, 5.6 Price Book, 5.7 Deal Optimizer, 5.8 Showroom Displays, 5.9 Labels, 5.10 Deliveries, 5.11 Warranty, 5.12 Marketing Hub, 5.13 (skipped — blocked on M04+M05), 5.14 Competitive Pricing, 5.15 Decision Engine, 5.16 Calendar. 6.9 Demand Forecasting | When real sales-line history doesn't exist yet (no Windward live, no quote_lines.sku field), PO-line qty over a fixed window (90d / 13 weeks) is the best available velocity proxy — at steady state, we re-order what we sell. Documented the assumption inline AND in the Daily Brief banner so future-me (or future user) doesn't mistake forecasted velocity for measured sales velocity. | When shipping a forecasting/derived-metric module ahead of the ideal data source, name the proxy explicitly in the UI ("velocity is derived from PO lines …") rather than presenting it as ground truth. Costs 1 extra sentence; saves the user trusting bad forecasts. When the real data lands (Track 6.11 here), swap the input function and leave the UI unchanged.
6.9 Demand Forecasting | Heuristic constants (DEMAND_LEAD_WEEKS, DEMAND_SAFETY_WEEKS, DEMAND_TARGET_WEEKS, DEMAND_OVERSTOCK_WEEKS) declared at module top. Five-line block, immediately tunable without hunting through the function. Same pattern as 5.7 Deal Optimizer's heuristic constants. | When a pure-compute module has tunable thresholds, declare them as ALL_CAPS const at the top of the file. Don't bury inside computeXxx(). Future tuning takes 10 seconds, and a reader can see the model assumptions at a glance.
6.9 Daily Brief tile gating | Reorder-Now tile fires only for senior roles even though the underlying alert exists for everyone — purchasing decisions belong to senior, warehouse already has the existing Low Stock tile (from reorder_point). Two near-duplicate tiles for different audiences > one tile that's wrong for half the users. | Daily Brief tiles should be tuned per role even when the data is universal. Match the tile to who will act on it; surfacing irrelevant items dilutes attention.
Global Search | Treated search as a meta-feature: every module already shipped its own `openX(id)` detail handler, so global search just had to map (type → icon, label-fn, subtitle-fn, openX-fn) and dispatch. ~270 LOC for cross-module search across 16 datasets. The handler-name mismatches that almost shipped (openJob vs openJobEdit, openArticle vs openArticleEdit, openDisplayEdit vs openShowroomEdit) only surfaced via a final grep audit before commit. | When a feature dispatches to many cross-module functions, run a single grep audit AFTER all the wiring is written but BEFORE commit. `grep -n 'function openX' index.html js/*.js` for each handler name confirms it actually exists. Cheaper than chasing runtime "X is not a function" later.
Topbar real estate | Added the search button between bell + page-actions. Three small UI elements visible in topbar (bell, search, contextual page actions) without crowding because the page-title sub-line + bell counter act as visual anchors. Order matters: search button styled as a fake input with placeholder text reads as a search field at a glance, not as another emoji button. | Treat search as a primary nav element, not a hidden keyboard shortcut. Visible search input style > 🔍 emoji button. The ⌘K hint baked into the button serves as discoverability for power users without forcing everyone else to memorize a shortcut.
Reports Center | 19 CSVs in one page took ~20 minutes total. Pattern: a single `collectReportDefs()` returns `[{key, label, description, count}]`, render iterates, `buildReportRows(key)` switches on key. New report = add 1 def + 1 case. Quote line items + PO line items shipped as their own separate reports rather than nested-JSON inside parent rows because Excel/Sheets users expect flat tables. | When a feature is "expose every dataset", design the registry pattern from line 1 — even if you only have 3 datasets at first. Adding the 4th, 5th, …, Nth report becomes a 4-line touch instead of a copy-paste snowball.
Reports CSV format | Used RFC 4180 quoting consistently: only quote when field contains comma/quote/newline; escape embedded quotes by doubling. Same code three times now (5.3 Inventory CSV import, 6.9 reorder export, Reports Center). | Promote `_toCsv(rows)` to a shared util in the next session. Inline triplication is fine until the fourth time; at that point extract.
