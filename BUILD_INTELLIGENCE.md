# BUILD_INTELLIGENCE.md — Lessons from each shipped item

> Append-only, one line per item. Format: `[item] | [gotcha or inefficiency] | [better approach]`
> Read this before starting a new build session — every entry makes the next session smarter.

0.2.B Settings Users panel | The legacy Settings card referenced removed `USERS{}` global → would have thrown ReferenceError in production if not caught. Lesson: when removing a global, grep ALL render functions, not just the auth block. | After deleting any top-level const/let, run `grep -n "<NAME>" *.html *.js` and fix every reference in the same patch.
0.2.B Settings Users panel | Inline `onclick="${onChange}"` with arrow-function IIFE shadows `this`, so the radio's value isn't accessible. Same pattern was bitten earlier in the score-state radio fix. | Default to a named global function (`onScoreStateChange(this, key)`) for any inline handler that needs `this`. Never use arrow IIFE in inline onclick/onchange.
2.1 Parent UI | When extending a `<label>` with sibling buttons inside a parent `<div>`, the surgical str_replace can leave a stray `</label>` orphan. Recovered by adding a hidden dummy `<label>` for balance — works but ugly. | Read 2 lines past the old_string when planning HTML edits to ensure tag balance. Or open the smallest possible block (the whole `<label>...</label>` pair) so the patch is self-balancing.
0.2.C RLS SQL | Dropping then recreating policies in a single SQL transaction works fine in Supabase, but if you forget the `DROP POLICY IF EXISTS` first, re-runs fail with "policy already exists". | Always lead each policy block with `DROP POLICY IF EXISTS`. Makes the file safely re-runnable.
0.4 Core schema | Naive 18-table file would have ~80 RLS policies if expressed inline. Used DO $$ … FOREACH … EXECUTE format() $$ to apply identical patterns in batches — collapsed to ~30 policies + 4 dynamic blocks. | For repetitive RLS patterns, use plpgsql DO blocks with arrays of table names. Halves the SQL size, makes patterns visible at a glance.
1.3 Daily Brief | "Build everything autonomously" runs out of runway when subsequent items all need M02 tables. Track 1.3 phase 1 used only existing data; phase 2 (quotes/pipeline tiles) is genuinely blocked. | When a track has multiple data dependencies, split it explicitly into "phase 1 (current data)" and "phase 2 (after M##)" so the unblocked work ships immediately and the blocked work has a clean re-entry point.
1.3 Daily Brief | Inline `onclick` strings using setTimeout (e.g. `goTo('vendors');setTimeout(...)`) feel hacky but are the cleanest way to navigate then update sub-tab state since renderVendors mutates the DOM the sub-tab variable references. | Tolerable pattern for now. Long term: refactor goTo() to accept a sub-tab arg (e.g. `goTo('vendors', 'scores')`) so callers don't need setTimeout sequencing.
1.1 vendor_scores | Same pattern as vendor_score_states (paginated load + on_conflict upsert). The first time the schema and the JS shape mostly match it goes fast; here it took ~5 minutes to ship. | When extending an existing persistence pattern (X_states → X_scores), copy-paste the loader/saver and rename — don't reinvent.
1.2 Quotes | Schema's quotes.notes is TEXT, but JS quote has 4 extras (contact/type/sqft/budget) the schema doesn't model. Stored as JSON string in notes — works but means notes is double-purposed. | Either extend schema with first-class columns OR move extras to a sidecar quote_attributes table. Acceptable for now since extras are sparse.
1.2 Quotes | Replacing all quote lines on save = DELETE + INSERT. Two roundtrips, but simpler than diffing. Edge case: if INSERT fails after DELETE succeeds, lines vanish. | For high-volume editing: batch into a single RPC. For our ~10-line quotes it's fine. If users start editing 100+ line quotes, revisit.
2.3 Coop tracker | Embedded inside Vendor Ranking page as a sub-tab — lower visibility than top-level nav, but doesn't add nav clutter. Daily Brief tile compensates by surfacing deadlines at the top of the dashboard. | Pattern works: secondary modules nest as sub-tabs; Daily Brief is the visibility layer that pulls them up when action is needed.
1.5 Pipeline | Old JS stages (prospect/quoted/ordered/complete) didn't map cleanly to schema (lead/qualified/quoted/negotiating/won/lost/abandoned). Did a hard refactor — old saved state in localStorage will look empty after deploy. Acceptable since pipeline was hardcoded demo data, not real production data. | Future schema/JS mismatches: if real data exists, write a migration. If demo only, hard-cut and note it in SESSION_LOG.
1.5 Pipeline | 8-factor probability model used heuristic numbers (e.g. referral=0.85). Will be wrong until probability_model_log accumulates real win/loss data and recalibrates. | Ship with documented heuristics. The recalibration job is its own future task — for now, the model produces sensible relative ordering even if absolute % is off.
4.1 Mgmt restructure | Replaced 70-line static "Pending Decisions" block with a sub-tab router (Overview/KPIs/Goals/Team Activity/System) — only ~90 lines net but unlocks 4 distinct screens. Pattern: when a page evolves from static to multi-screen, refactor to a sub-tab dispatch immediately rather than letting it grow as a single el.innerHTML. | If a render function exceeds ~100 lines or grows more than 2 distinct concerns, split into sub-tabs via a small `mgmtSection` global + dispatcher. Avoids the "one giant function" trap.
4.2 KPI auto-seed | First Owner load auto-seeds kpi_definitions if empty. Saves a manual M-task but creates a hidden side effect (data inserted via UI render). | Acceptable for owner-only seeding. Don't pattern-match this for non-owner cases — silent inserts on render are otherwise a smell.
4.2 KPI snapshot conflict | Used on_conflict=kpi_key,snapshot_date so re-running "Snapshot today" overwrites the same day's row instead of erroring. | When a user-triggered persistence action could be repeated in a session, design the unique key + on_conflict from the start.
4.3 Goals tree | Recursive renderGoalNode() with depth indent works fine for the expected goal counts (10-50). For 500+ goals, switch to virtualized rendering or lazy-expand on click. | Knowing the data scale lets you write the simpler implementation first.
3.2 Role dashboards | Three-branch role gate inside dashboard() (Warehouse / Sales / senior). Adding a 4th role variant means a 4th branch. | When role variants exceed 3, refactor to a roleDashboards = {Warehouse: fn, Sales: fn, ...} map with a default. Cleaner than nested if/else.
DOC-ONLY task batching (this session) | First three sessions interleaved doc updates between every code edit — wasted tokens re-reading file state. This session: code committed first, doc updates batched into single end-of-session commit. Saved meaningful tokens. | NEW RULE adopted: doc-only edits (BUILD_PLAN check-offs, SESSION_LOG, BUILD_INTELLIGENCE, MASTER edits) batched into one final commit per session, separate from code commits. Code commits stay focused.
