# AGENT MEMORY AND CONTEXT STRATEGY
> **Status:** Draft — Session 4 orchestration design
> **Goal:** End the "every session starts from zero" tax. Make architectural memory durable, canonical, and resistant to drift.

---

## 1. THE PROBLEM

Each agent session today rebuilds context by re-reading the entire docs tree. This is:
- Slow (token cost grows with repo size)
- Lossy (agents miss cross-file invariants)
- Drift-prone (two agents reading the same docs reach different conclusions)
- Operator-dependent (Michael fills the gaps verbally)

We need three layers of memory: **canonical**, **architectural**, **session**.

---

## 2. THREE-LAYER MEMORY MODEL

### Layer A — Canonical (slow, authoritative, rarely changes)

Single sources of truth. If two docs disagree, the canonical doc wins.

| File | Owns |
|---|---|
| `MASTER.md` | Business model, accounts, vision, hard rules |
| `BUILD_INTELLIGENCE.md` | Lessons learned, anti-patterns, gotchas |
| `AI_INTERACTION_MAP.md` | Worker ↔ frontend call paths |
| `MODULE_DEPENDENCY_AUDIT.md` | Module graph |
| `STARTUP_DEPENDENCY_ORDER.md` | Boot order invariants |
| `module_modes.json` | Registry truth (machine-readable) |

**Rule:** Canonical docs are append-mostly. Edits require an explicit relay packet with `corridor: docs-only` and the rationale in the commit body.

### Layer B — Architectural (medium-cadence, decision history)

`.orchestration/decisions/ADR-NNNN-*.md` — one file per architectural decision.
Format: context, decision, consequences, supersedes.

Why: prevents agents from re-litigating settled questions ("should we use Supabase or DynamoDB?") every session.

### Layer C — Session (fast, ephemeral, per-run)

| File | Lifetime |
|---|---|
| `WORK_IN_PROGRESS.md` | Overwritten every step |
| `PROMPT_LOG.md` | Append-only per session |
| `SESSION_LOG.md` | Append-only, batched at session end |
| `.orchestration/relays/*.json` | Per-task, expires |

---

## 3. CANONICAL REFERENCES (`CANON.md`)

Introduce a single index file: `CANON.md` at repo root. It lists, in order:
1. The 6 canonical files above with one-line purpose
2. The current ADR index
3. The "if you read nothing else, read these" list (max 3 docs per role)
4. A hash of each canonical file (regenerated by a pre-commit hook)

Agents read `CANON.md` first. If a referenced hash mismatches actual file content, the agent MUST re-read that file before acting. This catches stale agent caches.

---

## 4. PERSISTENT CONTEXT HANDOFF

Between sessions, context is carried by **the repo itself**, not by chat history.

### 4.1 Session start handoff
The auto-execute block in `.claude/CLAUDE.md` already does steps 1–7. Extend it to also:
- Read `CANON.md` and verify hashes
- Read the most recent 3 entries of `.orchestration/decisions/` index
- Read any `.orchestration/relays/*.json` where `to_agent` matches self and state is `open`
- Read any `.orchestration/escalations/*` that have not been resolved

### 4.2 Session end handoff
At clean pause, the agent writes:
- Updated `WORK_IN_PROGRESS.md`
- `SESSION_LOG.md` entry (batched commit)
- If applicable, a new ADR in `.orchestration/decisions/`
- Updated `BUILD_INTELLIGENCE.md` if a new lesson emerged
- `efficiency-log.md` (per existing efficiency-monitor skill)

### 4.3 Mid-session checkpoint
For long sessions, every N commits the agent writes a `.orchestration/checkpoints/<branch>.json` with: last good sha, current sub-goal, files touched, open questions. Crash recovery reads this.

## 5. ARCHITECTURAL MEMORY (ADRs)

Lightweight ADR template:

```markdown
# ADR-0007: Use Cloudflare Worker as Anthropic proxy

**Date:** 2026-05-15
**Status:** Accepted
**Agents involved:** claude-code, michael

## Context
Frontend cannot safely hold ANTHROPIC_API_KEY...

## Decision
Route all Anthropic calls through accentos-anthropic-proxy.mgraf77.workers.dev.

## Consequences
- (+) Key never leaves Cloudflare secrets store
- (+) Centralized rate limiting and CORS
- (−) Adds one hop of latency
- (−) Worker version drift becomes a new failure mode

## Supersedes
None.

## Referenced by
AI_INTERACTION_MAP.md
```

Numbering is monotonic. ADRs are never deleted, only superseded.

## 6. ANTI-DRIFT MECHANISMS

### 6.1 Hash-pinned canonical refs
`CANON.md` carries SHA256 of each canonical doc. Pre-commit hook fails if canonical files changed without `CANON.md` updating. Agents that read a canonical file MUST verify hash.

### 6.2 Single-writer invariant per doc
Each canonical doc declares its writer agent (in a frontmatter line). Other agents may propose edits via relay packet but cannot directly commit.

### 6.3 Drift detector
A scheduled job (or session-start check) runs `scripts/drift-check.sh`:
- Verifies CANON.md hashes
- Flags ADRs marked `Proposed` older than 7 days
- Flags `WORK_IN_PROGRESS.md` last-updated > 14 days (stale resume risk)
- Diffs `module_modes.json` against `MODULE_MODES.md` table

Output lands in the Control Plane dashboard.

### 6.4 Forbidden-edit list
A `.orchestration/forbidden.json` declares files that no agent may edit without explicit Michael approval (e.g. `wrangler.toml`, anything under `sql/migrations/<deployed>/`). Pre-commit hook enforces.

---

## 7. SESSION CONTINUITY ACROSS AGENTS

When Agent A hands to Agent B mid-feature:
1. Agent A reaches clean pause (see Handoff Protocol §7).
2. Agent A writes a relay packet with `context_refs` listing the minimal set of files B must read.
3. Agent B starts session, reads `CANON.md` + the relay packet's `context_refs` + `WORK_IN_PROGRESS.md`. That should be sufficient — if B needs more, it's a signal that A's packet was incomplete (logged for protocol improvement).

The goal: minimum-viable context, not maximum context. Reading the whole repo every time is failure, not safety.

---

## 8. MEMORY HYGIENE

- Logs > 90 days old are archived under `docs/archive/<year>/`
- ADRs are never archived
- `PROMPT_LOG.md` rolls over per quarter
- `BUILD_INTELLIGENCE.md` is reviewed monthly; obsolete lessons get a `superseded-by:` line, not deletion

---

## 9. WHAT WE EXPLICITLY DO NOT DO

- No vector DB / RAG layer yet. The repo is small enough that file-based canon + targeted reads beats embedding drift.
- No agent-private memory stores. Everything an agent "remembers" must be in the repo. If it's not in the repo, it didn't happen.
- No cross-session chat replay. Context is reconstructed from artifacts, not transcripts.
