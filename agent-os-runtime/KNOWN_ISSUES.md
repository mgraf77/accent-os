# Known Issues — AgentOS Runtime v1
**As of:** 2026-05-08

---

## Active Issues

### KI-001 — No Task Status Writeback
**Severity:** Medium
**Impact:** The runtime evaluates tasks but cannot change their status. After a tick runs and determines TASK-000003 is "ready", the task file still reads `status: pending`. A human or external process must update the markdown file manually to advance state.
**Workaround:** Manually edit task frontmatter `status:` field.
**Fix:** Build `runtime/task_writer.py` (see NEXT_STEPS.md).

### KI-002 — Event Log Counter Not Cross-Session Atomic
**Severity:** Low
**Impact:** EVT-XXXXXX IDs are generated by counting existing events in today's log file. If two processes run simultaneously, they could produce duplicate event IDs.
**Workaround:** Run one tick at a time (current design is single-process).
**Fix:** Switch to UUID-based event IDs, or use a file lock around event writes.

### KI-003 — queue.json Not Write-Safe Under Concurrency
**Severity:** Low
**Impact:** `state/queue.json` is written with a full overwrite. Concurrent writers would corrupt it.
**Workaround:** Single-process operation only.
**Fix:** Use atomic write (write to temp file, then rename) or SQLite.

### KI-004 — No Validation of `depends_on` References
**Severity:** Low
**Impact:** A task can declare `depends_on: [TASK-999999]` referencing a non-existent task. The dependency engine will treat it as perpetually unsatisfied (blocking the task), but will not surface an explicit "dangling reference" error.
**Workaround:** Manually verify task IDs in `depends_on` lists.
**Fix:** Add a cross-reference validation pass in `validate_task.py`.

### KI-005 — `docs/` Directory is Empty
**Severity:** Informational
**Impact:** The `docs/` directory was created but no architecture documentation was written beyond README.md and these state files.
**Fix:** Populate during v1.1 or governance phase.

---

## Non-Issues (by design)

- **No auto-execution** — tasks are queued but not executed. Intentional v1 constraint.
- **No network calls** — the runtime is fully local. Intentional.
- **critical tasks always blocked** — TASK-000004 is blocked by design; this is correct behavior.
