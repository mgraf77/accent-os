# ADR-001: Supabase as primary database

**Status**: Locked  
**Date**: 2026-04 (early AccentOS build)

## Decision

Use Supabase (PostgreSQL + Row-Level Security + Auth) as the sole database and authentication layer for AccentOS. AccentOS enforces customer mode data isolation using Supabase row-level security policies.

## Context

AccentOS needs a persistent backend for: vendor scores, customer CRM, quotes, pipeline, purchase orders, and all CRUD modules. Needs auth with 5 roles (Owner, Admin, Manager, Sales, Warehouse).

## Rationale

- PostgreSQL with RLS lets each row be secured per-role without application-layer auth logic
- Supabase REST API (PostgREST) = zero custom backend code; direct fetch() calls from JS
- Free tier covers AccentOS scale (< 10 users, < 500MB)
- Built-in auth with JWT; anon key embedded for public Cloudflare Pages deploy
- Supabase MCP server available for Claude Code (AI can run SQL directly)

## Consequences

- All schema changes require SQL migration files run by Michael (M## tasks)
- No real-time subscriptions needed (pull-on-navigate pattern)
- JWT auth means offline mode is limited to localStorage-backed features
- pgvector extension available for M42/M43 RAG phase

## Alternatives considered

- Firebase: rejected (NoSQL, weaker relational queries)
- PlanetScale: rejected (no RLS, MySQL)
- Self-hosted PostgreSQL: rejected (operational burden)
