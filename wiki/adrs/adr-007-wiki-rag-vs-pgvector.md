# ADR-007 — Wiki-Pattern RAG vs pgvector

## Status

Accepted — wiki-pattern shipped v6.11.1 (2026-05-06)

## Context

AccentOS needed a knowledge retrieval system so Claude Code sessions could reference Accent Lighting domain knowledge (SOPs, product clusters, employee roles, data sources, architectural decisions) without re-explaining context in every prompt.

Two approaches were evaluated:

### pgvector approach (prior)
- Store wiki pages as text chunks in a `knowledge_embeddings` Postgres table.
- Embed queries + documents using an embedding API (e.g. OpenAI `text-embedding-3-small`).
- Retrieve top-K chunks by cosine similarity (`<=>` operator).
- Pro: semantic recall — finds "luminaire" when query says "fixture".
- Con: embedding API cost ($0.02–0.05/1K queries), re-embed on every page edit, pgvector extension setup in Supabase, latency ~400–600ms/query.

### Wiki-pattern approach (chosen)
- Store wiki pages as flat `.md` files in `wiki/` subdirectories.
- Retrieve by keyword overlap (TF-IDF approximation) against title + tags + body.
- Boost by per-page `weight` multiplier.
- Index generated by `wiki_seed.py` into `_index.md` + `_index.json`.
- Pro: free, instant (~50ms), no API calls, editable in any text editor, version-controlled.
- Con: no semantic similarity — misses synonym queries unless tags are enriched.

## Decision

Wiki-pattern. The operational simplicity and zero-cost advantage outweigh the recall gap.

**Eval matrix (full detail in `skills/accent-rag/eval-matrix.md`):**

| dimension | wiki-pattern | pgvector | delta |
|---|---|---|---|
| Recall | 7/10 | 9/10 | -2 |
| Precision | 8/10 | 8/10 | 0 |
| Latency | 10/10 (~50ms) | 6/10 (~500ms) | +4 |
| Cost | 10/10 (free) | 5/10 ($$/query) | +5 |
| Maintenance | 9/10 (edit files) | 5/10 (re-embed) | +4 |
| **Total** | **44/50** | **33/50** | **+11** |

## Recall gap mitigation

The -2 recall gap is addressed by:
1. **Synonym tags**: all cluster pages carry synonym aliases (`luminaire`, `fixture`, `light-fitting`) so keyword search catches them.
2. **Query enrichment** (future): wiki_seed.py can expand queries with synonym lists before scoring.
3. **Concept pages** (future): high-level concept pages cross-link to specific SOPs/ADRs, acting as query bridges.

## Consequences

- **Positive**: Claude Code sessions can search the wiki instantly with no external API calls.
- **Positive**: Michael can add/edit wiki pages in any editor; `wiki_seed.py` regenerates the index in seconds.
- **Positive**: All knowledge is version-controlled in the same repo as the code.
- **Negative**: Recall is lower for semantic queries. Mitigated by tag enrichment.
- **Migration path**: if recall gap becomes a real problem after the wiki grows to 100+ pages, pgvector can be added as a second-pass re-ranker without changing the wiki format.
