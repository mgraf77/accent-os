# Session Reset Resilience — AccentOS

> Doctrine only. No runtime, no daemon, no substrate claims.
> Branch: `claude/orchestration-maturity-analysis-qdJ5W`
> Date: 2026-05-10
> Companion to `RELAY_COMPRESSION_PROTOCOL.md` and `SELF_CONTAINED_EXECUTION_WINDOWS.md`. Read those first.

---

## 0. Frame

Session reset resilience is the property that when a Claude session ends — for any reason — the next Claude session can be picked up by Michael with bounded re-orientation cost. The end may be planned (clean-freeze) or unplanned (Codespace stop, OOM, network drop, model usage limit, mobile app close).

This is **resilient procedural orchestration**, not durable runtime orchestration. The distinction is sharp and load-bearing — see §10. Failing to honor it is the most common pathway to fake-runtime claims.

---

## 1. Two failure modes the doctrine targets

### 1.1 — Planned end
Session ends because a packet's exit condition tripped. Clean-freeze acquisition (`RELAY_COMPRESSION_PROTOCOL.md` §8) ran, an artifact was written, a next-packet draft was generated. Resilience here is: how *fast* can Michael read the artifact and continue the chain.

### 1.2 — Unplanned end
Session ends because of an external event the runtime cannot observe and the doctrine cannot prevent. Resilience here is: how much state was *checkpointed* before the end, and what does the next session's auto-execute sequence (`.claude/CLAUDE.md`) recover from disk.

The doctrine treats (1.1) as the optimization case and (1.2) as the *constraint* case. The constraint case sets the floor; the optimization case sets the ceiling.

---

## 2. Clean freeze acquisition (recap, with the resilience lens)

Clean freeze (per `RELAY_COMPRESSION_PROTOCOL.md` §8) succeeds when the in-flight unit of work is finished or reverted, the resume artifacts on disk are accurate, and the closing turn output is scannable.

Adding the resilience lens, three additional rules:

- **No half-edits at any moment.** A session that ends mid-edit corrupts resume. Edits must be applied as one Edit call per file or batched into a single commit; never spread across multiple turns without a commit gate.
- **Commit before declaring freeze.** A clean freeze without a commit leaves untracked changes that the next session may not recognize as "the work the artifact describes."
- **WIP file is updated last.** `WORK_IN_PROGRESS.md` is the canonical resume entry per `.claude/CLAUDE.md`. Update it after the commit, not before — so its content matches the commit it describes.

If any of those three is violated, the freeze is not clean even if the artifact text says it is.

## 3. Resume packet generation

The resume packet is the artifact a future session reads to know what was happening. At L1 it is constructed from the union of:

- `WORK_IN_PROGRESS.md` — the human-readable resume baton
- The clean-freeze artifact text in the closing turn (Michael keeps in the chat)
- The latest commit on the branch (the durable record of what work landed)
- `SESSION_LOG.md` last entry (the long-form narrative)

The resume packet is *generated by Michael at relay time* by selecting from those sources. Doctrine does not centralize this into one file — that would be an L2 step, requiring durable structured state (`ORCHESTRATION_BOTTLENECK_MAP.md` B2) which does not exist.

What doctrine *does* require: each of the four sources is kept in a state where Michael can extract the needed parts in <60 seconds. Specifically:

- `WORK_IN_PROGRESS.md` — current to last sub-step, not "current as of yesterday"
- Clean-freeze artifact — present in the closing turn whenever a freeze occurred
- Latest commit — its message describes what landed (no "wip"/"misc"/"updates")
- `SESSION_LOG.md` — last entry summarizes the session including freeze reason if applicable

Source-fidelity is the resilience. Centralization is L2.

---

## 4. Checkpoint doctrine

A checkpoint is a moment when on-disk state accurately reflects in-progress work. Checkpoints are not separate from commits — at L1 the commit *is* the checkpoint.

Required checkpoint discipline:

- **Per sub-step.** Sub-step finishes → file edits saved → `WORK_IN_PROGRESS.md` updated. This is the *minimum* checkpoint.
- **Per commit.** Commit lands → `WORK_IN_PROGRESS.md` reflects the post-commit state → `SESSION_LOG.md` only updated at session end (per OPERATING RULES batched-doc-update).
- **Per packet exit.** Packet ends → clean-freeze artifact written → final commit lands → `WORK_IN_PROGRESS.md` reflects "session paused" with resume hint.

There are no checkpoints between sub-steps (sub-sub-step granularity is a working-memory concern). Edits that span multiple turns without a sub-step boundary are forbidden under this doctrine — they create a checkpoint gap an unplanned end can fall into.

## 5. Restart recovery

The next session's auto-execute sequence (`.claude/CLAUDE.md` AUTO-EXECUTE) is the recovery mechanism. Doctrine does not extend it; doctrine ensures it has good inputs.

The sequence today:
1. Activate vibe-speak in default mode
2. Log session start
3. Read `WORK_IN_PROGRESS.md`
4. Read `BUILD_PLAN_CLAUDE.md` for next item
5. Read `BUILD_INTELLIGENCE.md`
6. Run `scripts/status.sh`
7. Begin work

Restart-recovery resilience requires steps 3 and 4 to be *trustworthy without further inference*. The doctrine's contribution: clean freezes write WIP and SESSION_LOG in a shape step 3 can use directly.

What restart recovery does NOT do today (and what doctrine does not add):

- Does not parse a structured state file as authoritative (L2)
- Does not reconcile with an external supervisor (L3)
- Does not re-run a failed sub-step automatically (L5)

If any of those is wanted, it is a maturity step, not a doctrine refinement.

## 6. Session exhaustion handling

Session exhaustion is the class of failure where the session ends because it ran out of one of: token budget, model usage allowance, context window, mobile app foreground. None are observable to the runtime in advance with high precision. All are observable as *approach signals*.

Doctrine for handling approach signals:

- **Pre-emptive freeze on context approach.** When a session's context appears to be reaching its limit (long turns, many large reads), the next sub-step boundary is a freeze, not a continue. The default packet authority list does **not** authorize Claude to push into the last 10% of context — freeze is mandatory.
- **Pre-emptive freeze on commit-cap approach.** If the packet has a commit cap (per `SELF_CONTAINED_EXECUTION_WINDOWS.md` §2.3) and the next iteration would land at the cap, freeze before that iteration, not after.
- **Pre-emptive freeze on suspected usage threshold.** Claude does not see its model usage allowance directly; if signs of throttling appear (slow responses, retries on assistant turn), treat it as exhaustion approach and freeze.

The principle: exhaustion is dignified when it is anticipated and ugly when it is met. Doctrine biases toward early freeze.

A note on token-budget specifically: at L1 there is no reliable token meter (`ORCHESTRATION_BOTTLENECK_MAP.md` B3). Pre-emptive freeze on token grounds is a heuristic, not a measurement. That is fine — heuristic-driven freeze is conservative; the worst case is one extra relay.

---

## 7. Usage-threshold handling

Distinct from exhaustion: usage thresholds are policy-level limits (e.g., "do not spend >$X on this packet") that the packet author may set. Today no automated enforcement exists. Doctrine therefore treats usage thresholds as *self-honored caps in the packet*:

- **Stated caps in the packet.** "Stop after 5 commits" or "stop after 1h wall time" are stated by Michael in the opening relay.
- **Self-reporting at exit.** The clean-freeze artifact reports *which* cap tripped, not just that one tripped. "Hit commit cap (5/5) at HH:MM" is the resolution.
- **No silent extension.** A packet whose cap trips does not "just one more" itself. The cap is the cap.
- **No retroactive widening.** A subsequent packet may extend the work, but the prior packet stays exited at the cap. Doctrine does not allow rewriting the freeze artifact to widen the original cap.

This pattern is the working substitute for an L3 supervisor's budget enforcement. It is honest only because the limits are author-stated and Claude-honored — not infrastructure-enforced. Doctrine names this so that future readers do not mistake it for the real thing.

## 8. Continuation preservation across sessions

Continuation is preserved when the next session can resume the *intent* of the prior packet without Michael re-explaining it. The mechanism is the next-packet draft generated at clean-freeze (`RELAY_COMPRESSION_PROTOCOL.md` §9), pasted by Michael into the new session as the opening relay.

What is preserved:
- The packet shape (scope, exits, authority, escalation triggers)
- Knowledge of what landed in the prior session (commit references)
- The freeze reason
- The suggested next iteration

What is *not* preserved:
- Working memory (variables, intermediate findings not committed to disk)
- Tool state (open file descriptors, in-progress agent calls, partially-applied permissions)
- Anything held only in the chat history of the prior session

The latter list is the L1 ceiling. Pretending it crosses session boundaries is fake-runtime behavior. The former list is what the next-packet *carries* in its body.

A successful continuation looks like: Michael pastes a packet → fresh session reads it + WIP + last commit → resumes at the next sub-step described in the packet → produces commits that extend the prior session's commit chain → exits with another freeze artifact.

A failed continuation looks like: Michael pastes a vague "continue from last session" → fresh session reads WIP and tries to infer the intent → infers wrong → produces commits that diverge from the prior plan. The cure is *better packet writing*, not better runtime.

## 9. Specific failure modes and their resilience answers

| Failure | Resilience answer |
|---|---|
| Codespace stops at 30 min idle | Per-sub-step WIP updates ensure no more than one sub-step is lost; commit-before-WIP rule ensures the WIP describes a real commit |
| Mobile app closed mid-turn | Same as above; assistant turn is lost but the prior commit + WIP survive |
| Anthropic API rate limit | Pre-emptive freeze on suspected throttling (§6); next session retries from WIP |
| Model usage allowance hit | Same as rate limit; freeze pre-emptively when responses degrade |
| Network blip during `git push` | Standard 4-retry exponential backoff; if all four fail, freeze with WIP describing the unpushed commit |
| Pre-commit hook fails | Single retry, then freeze (do not skip hooks) |
| Disk full | Freeze immediately; WIP write is the cap and if it fails, do not commit |
| Workspace deleted (Codespace rebuild) | Local resilience: none. Remote branch state is the resume source. Doctrine: every commit pushed promptly so no work lives only in a deleted workspace |
| Two sessions opened concurrently | Out of scope at L1 — doctrine does not protect against this; the L4 row of the maturity model is required. Recommended workaround: procedural ("only one session at a time") |
| Packet's premise turns out wrong | Freeze with escalation; do not iterate past a wrong premise |

Failures with no doctrine answer (real but out of scope at L1): silent corruption of `WORK_IN_PROGRESS.md` between sessions, Git history rewrite by an external actor, branch deleted on remote. These require L2+.

## 10. Resilient procedural orchestration vs. durable runtime orchestration

This is the section that matters most. Mistaking one for the other is the doctrine's central error mode.

### Resilient procedural orchestration *(this document)*

- **Driver:** human-authored relays plus session-start auto-execute steps
- **State:** files in the repo (WORK_IN_PROGRESS, SESSION_LOG, commits, branch); no special database
- **Recovery:** human reads files, composes next packet, pastes
- **Failure model:** session death loses the in-flight assistant turn; commits + WIP survive
- **Bounded by:** human relay frequency, per-packet authority breadth, file-fidelity discipline
- **Honesty test:** can be turned off entirely by Michael not relaying — there is no process to "leave running"
- **Maturity level:** L1 (per `ORCHESTRATION_MATURITY_MODEL.md`)

### Durable runtime orchestration *(NOT this document; NOT being built)*

- **Driver:** a process outside any single Claude session that holds lifecycle, state, and decisions
- **State:** structured records in a store the runtime owns; markdown is a rendered view, not authoritative
- **Recovery:** runtime detects session death and either restarts or escalates without human relay
- **Failure model:** session death loses nothing; the runtime resumes
- **Bounded by:** infrastructure properties (claim arbitration, supervisor liveness, escalation channel)
- **Honesty test:** behaves the same when Michael is asleep
- **Maturity level:** L2 / L3 / L4 / L5 depending on which capabilities are present

### How to tell which one a pattern is

Apply this question: *if Michael never sends another message, what continues to happen?*

- **Resilient procedural:** nothing happens. The system goes quiet. This is correct and safe.
- **Durable runtime:** work continues per the runtime's plan. This is what AccentOS does not have.

If a doctrine pattern claims the system "continues without Michael," the pattern is misclassified. Procedural resilience is about *the next session starting cleanly*, not about *something running in between*.

---

## 11. The single sentence

Session reset resilience makes the next session start cheap; it does not make the gap between sessions productive.
